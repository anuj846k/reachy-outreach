# Kakiyo — AI-Powered Outreach Platform

Kakiyo(reachyy) is a full-stack AI outreach tool. You paste a LinkedIn URL or any public profile, it extracts a structured prospect profile in the background, and generates a personalized outreach message. It also handles conversation follow-ups with full context memory.

---

## Architecture Overview

```
Browser → Next.js Server Action → Neon Postgres
                              ↓
                         Inngest Event
                              ↓
                    Background Function
                    ├── Apify (LinkedIn)
                    ├── Firecrawl (other URLs)
                    └── Gemini (consolidation)
```

## Technical Decisions

### 1. Middleware is not the auth gate

Next.js middleware runs on the **Edge Runtime** — no database access, no Node.js APIs. It behaves like a proxy: it can redirect and set headers, but it cannot verify a session against the database.

I deliberately did not rely on middleware as the sole auth layer. If the middleware check is bypassed or misconfigured, every data route would be exposed.

**What I did instead:** Middleware only handles fast redirects — unauthenticated users away from `/dashboard/*`, logged-in users away from `/login`. Every Server Action and API route has its own explicit auth check via `requireAuth()` and `requireUnauth()` utility functions I wrote. These call `auth.api.getSession()` directly — hitting the database and getting the real session or throwing.

This means even if middleware is entirely disabled, every data operation remains protected at the server level.

---

### 2. Prospect extraction runs as a background job

Extracting a prospect profile involves calling Apify to scrape LinkedIn (3–8 seconds), Firecrawl to scrape other URLs (2–5 seconds each), and Gemini to consolidate everything (2–4 seconds). Total: 10–20+ seconds, unpredictable, and it can fail at any step.

Running this inside a request/response cycle would either time out or leave the user waiting with a frozen UI.

**I used Inngest** for the extraction pipeline. The Server Action inserts the prospect record with `status: pending` and fires an event. The HTTP request returns immediately. Inngest picks up the event and runs the pipeline as a durable background function.

I wrapped each operation in `step.run()` — a separate tracked unit. If step 3 fails, Inngest retries just that step with exponential backoff. Nothing runs from scratch on retry.

---

### 3. `step.ai.wrap()` for Gemini calls inside background jobs

I needed to call Gemini inside the Inngest function to consolidate scraped data into a structured profile. A plain `generateText()` call inside `step.run()` works, but you lose all observability — you can't see what prompt was sent, the response, token usage, or trace retries.

**I used `step.ai.wrap()`** instead. This keeps the Vercel AI SDK `generateText` call unchanged and wraps it so Inngest treats the entire AI call as a first-class tracked step with full observability: prompt, response, token usage, latency, and retries — all visible in the Inngest dashboard.

**A bug I hit and fixed:** `step.ai.wrap()` returns a response object where `.output` is a JavaScript getter, not a plain value. Inngest serializes step results as JSON between steps — JSON strips getters. On replay, `result.output` is `undefined`.

I fixed it by falling back to `result._output`, which is the plain underlying property that survives serialization:

```ts
const extracted = result.output ?? (result as any)._output;
```

---

### 4. Outreach generation is synchronous — no Inngest

Message generation is a single Gemini call. The user is waiting for the result and needs immediate feedback. There are no multiple steps, no file processing, nothing that would time out.

**I kept `generateOutreach()` as a plain Server Action** that calls Gemini directly and returns the result to the client. Server Actions on Vercel can run for up to 60 seconds — more than enough for a single generation call.

Adding Inngest here would introduce polling overhead, webhook round-trips, and UI complexity for no benefit. I only use Inngest where the work is genuinely long-running, multi-step, or needs to survive server restarts.

---

### 5. `loading.tsx` and `error.tsx` for automatic UI states

Every page that fetches data has a corresponding `loading.tsx` and `error.tsx`. I built skeleton layouts that mirror the real page structure precisely — no layout shift when data arrives. `error.tsx` acts as an error boundary so a failed database query shows a recoverable UI instead of a white screen. No extra client-side logic needed.

---

### 6. `revalidatePath` as the cache invalidation bridge

Server Actions mutate the database but Next.js caches page data aggressively. Without explicit invalidation, a user could add an offering and see the old list on the next load.

**I call `revalidatePath()`** after every database write, targeting the affected routes. This marks them as stale so the next request re-fetches from the database.

---

### 7. Template-based prompt customization — no database table

I needed to let users customize the system prompt that drives message generation. The obvious approach is a settings page with separate form fields for tone, length, opening style, closing style, avoidance rules — stored in a `user_settings` table.

**I built a template string system instead.** The default prompt is a string constant with `{{placeholders}}`. Users can expand a collapsible section in the generation dialog, edit the textarea, and click variable badges to insert tokens at cursor position.

At generation time, I replace placeholders with real values and pass the filled string as the Gemini `system` prompt. No database table. No migration. No schema overhead. Customization is per-message, which is the right scope — every outreach message should be intentional anyway.

---

### 8. Rolling summary instead of RAG for conversation memory

When generating a follow-up reply, the AI needs context of the full conversation. One option was RAG — embed every message as a vector, store in a vector database, and on each reply retrieve semantically relevant past messages.

**I chose not to do RAG, and here's why:** RAG is designed for searching large unstructured knowledge bases — PDFs, documentation, thousands of documents. LinkedIn outreach conversations are linear and short — typically 10–20 messages before a meeting is booked or the lead goes cold. RAG retrieves isolated fragments out of chronological order, which destroys conversational flow. It also adds an embedding call and a vector search on every generation — significant overhead for zero benefit at this conversation length.

**I store conversation messages in order** in a simple `conversation_messages` table and pass the full thread as context when generating follow-ups. It's always within Gemini's context window, perfectly chronological, and costs nothing extra. If conversations grew very long, I'd add a background Inngest summarization step — but that never happens in practice for LinkedIn outreach.

## Database Schema

```
user                    (Better Auth managed)
session                 (Better Auth managed)
offerings               offering name, summary, ICP, differentiators
prospects               name, jobTitle, company, bio, painPoints, sources (jsonb)
outreach_messages       content, subjectLine, tone, status, rollingSummary
conversation_messages   role (user/assistant), content, FK → outreach_messages
```
